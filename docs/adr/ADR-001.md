# ADR-001: Codex TUI への完全自動 submit を1物理行フレーミングで実現する

## Status: Accepted

## Context
xrev は cmux のペイン間通信だけで AI 同士のレビュー往復を完全自動化する（人介在ゼロが必須要件）。送信は `cmux send` で本文を Codex(TUI) の入力欄にペースト → `cmux send-key enter` で送信 → `read-screen` で応答回収、という手順。実機検証で、短い1行プロンプトは送信できるが、長い複数行プロンプトはペーストされるのに Enter で送信されず入力欄に滞留する不具合が判明した。原因は、複数行ペーストの描画完了前に Enter が消費される/複数行入力中の Enter が改行扱いになること。これは xrev の自動化要件を直接損なうため、この設計往復(xrev 自身を Codex でクロスレビュー)で解決方針を決めた。

## Decision
送信本文を『画面上は1物理行・意味上は複数行』に変換して送る。全モード共通の不変条件は (1)cmux に実改行を渡さない=常に1物理行 (2)round_id 相関で完了判定 (3)eval/未クォート展開を使わずクォート済み引数で渡す。本文表現は content_type で二段構え: plain(散文)は改行を ASCII トークン <XREV-NL> に置換したコンパクト1行、unified_diff/code/markdown は番号付き line framing 『XREV_PAYLOAD_V1 round_id=<id> content_type=<ct> lines=<n> || L0001: ... || L0002: ...』(|| L[0-9]+: だけを行境界として解釈、行頭+/-とインデントを保持)。末尾に END_ROUND_<id> マーカーを置き切り詰めを検出。送信は send(長大なら Enter 無しでチャンク追記)→描画待ち→Enter1回。応答検出は round_id 一致の新着センチネル JSON。review-schema にトップレベル round_id(任意)を許容する。base64 と『\n エスケープを1行で送る』案は不採用(前者は LLM 復元が不安定、後者は cmux の自動アンエスケープで実改行化し再発する恐れ)。

## Consequences
利点: 任意長の複数行を人介在ゼロで確実に submit できる/diff の行構造(行番号・+/-・インデント)を保持しレビュー精度を保つ/round_id と END マーカーで送信完了と切り詰めを機械的に検証できる。妥協: LLM に line framing の軽い再構成を要求する(base64 ほどではない)/content_type 分岐で実装がやや複雑化/区切りトークン衝突や大 diff 分割は medium として実装・テストで対処/トークンの正典化前に cmux send の実挙動をスクラッチ検証する必要がある(|| や L0001: の素通り、バックスラッシュ展開、タブ/連続スペース潰れ、長大1行の切り詰め、Enter1回submit)。Codex も『設計として収束、critical/high の懸念なし、残りは medium』と判定。

## Discussion log
- [human/propose] xrev 自身の設計方針が妥当か @xrev でレビューしたい。そのレビュー送信中に長文 submit の不具合が顕在化した。
- [codex/react/high] TUI に複数行を作るな。本文を1物理行にエンコードして送れ(JSONエスケープ or base64)。Enter=送信、ソフト改行キーは脆い。完了は round_id 相関で判定。
- [claude/react] 1物理行化と round_id 相関に同意。ただし base64 は LLM 復元が不安定で却下。\n エスケープは cmux の自動アンエスケープで実改行化し再発しうるので、可視セパレータでの1物理行化を対案とする。
- [codex/react/medium] base64 却下に同意。Unicode 記号より ASCII トークン<XREV-NL>推奨。ただし単純な改行潰しは diff に弱い→番号付き line framing(L0001:|| 区切り+ヘッダ)を正規ルートに。
- [claude/react] line framing 採用に同意。ただし per-line len= は入れず最小に、散文は framing 省略可(content_type で軽重)、実装前にスクラッチで cmux 挙動を確定したい。
- [codex/react] len= 省略でほぼ無害。plain は compact、diff/code は framed の二段構えは妥当。共通不変条件(1物理行/round_id/実改行を送らない)は統一。スクラッチ検証の要点を提示。
- [codex/decide] 設計として収束。critical/high の懸念なし。残るは medium(大diff分割・content_type誤判定・トークン衝突・round_idスキーマ拡張)で実装/テストで潰せる範囲。
- [claude/decide] 収束を確認。番号付き line framing＋round_id＋描画待ち＋Enter1回＋チャンク＋END マーカーで実装する。base64/\nエスケープは不採用。
